# AI 项目 Review 规范

本文件定义 Codex、ChatGPT、Claude Code、Gemini CLI 等 AI Coding Agent 执行项目审查时的统一流程。

## 1. Review 目标

Review 不是泛泛点评，而是输出可定位、可执行、可验证、可追踪的问题清单。

每次 Review 应做到：

- 默认使用简体中文输出；
- 基于实际代码和项目文档，不凭经验补全项目；
- 区分已确认问题、潜在风险和待确认问题；
- 每个问题提供证据、影响、建议和验收标准；
- 将需要处理的问题同步到 `docs/todo.md`；
- 修复并验证通过后自动勾选对应待办；
- 无法验证时如实标记，不得假装完成。

## 2. Review 开始前

开始审查前，依次读取与本次任务相关的文件：

1. `README.md`
2. `AGENTS.md`
3. `docs/context.md`
4. `CONTEXT.md`
5. `docs/architecture.md`
6. `docs/decisions.md`
7. `docs/todo.md`
8. 当前任务涉及的源码、配置、测试和提交差异

然后确认本次 Review 范围：

- 全项目；
- 指定模块；
- 指定文件；
- 指定提交；
- 指定 Pull Request；
- 指定功能或缺陷。

如果范围不完整但仍能安全审查确定部分，可以继续；无法确认的内容必须标记为“待确认”，不得猜测。

## 3. 输出语言

- 所有分析、结论、待办、修改说明和总结默认使用简体中文。
- 代码、命令、文件名、类名、接口名、错误原文和通用技术术语可保留英文。
- 引用英文错误时，应补充中文解释。
- 不得无故输出大段英文内容。

## 4. 审查维度

至少检查以下内容：

1. 功能正确性；
2. 崩溃风险与异常处理；
3. 并发、线程和异步任务；
4. 内存、资源和生命周期；
5. 架构边界与模块职责；
6. 重复实现与可维护性；
7. 性能问题；
8. 安全、隐私和敏感信息；
9. 加载、空数据、错误和重试状态；
10. 平台兼容性；
11. 测试覆盖；
12. 文档与注释是否同步。

### ForumHub iOS 项目额外检查

- Swift 并发安全与 `MainActor` 使用；
- SwiftUI 状态管理或 UIKit 生命周期；
- iOS 版本兼容、真机与模拟器差异；
- 多数据源切换、频道状态和分页状态隔离；
- Scene 生命周期、Deep Link 和后台恢复；
- 网络失败、登录失效、Cookie/Token 恢复和图片加载异常状态；
- NGA、V2EX、LINUX DO 适配层是否泄漏到共享 View；
- 帖子连续分页、去重、楼层锚点和任务取消是否稳定。

## 5. 问题分级

每个问题必须同时标注优先级和可信度。

### 优先级

- `高`：可能导致崩溃、数据丢失、安全问题或核心功能不可用；
- `中`：影响稳定性、性能、维护性或重要用户体验；
- `低`：非关键体验、文档或长期维护问题。

### 可信度

- `已确认`：可以由当前代码、配置、测试或运行结果直接证明；
- `高概率`：有充分依据，但尚未完成运行验证；
- `待确认`：缺少必要上下文、配置或运行环境。

规则：

- 已确认问题可以直接修复；
- 高概率问题可以加入待办，但必须标记需要验证；
- 待确认问题只记录，不得直接修改或标记完成；
- 不得把推测描述为确定事实。

## 6. 单个问题格式

每个问题使用唯一编号，例如 `R-001`、`R-002`。

```md
### R-001：播放器地址存在强制解包

- 优先级：高
- 可信度：已确认
- 类型：崩溃风险
- 涉及文件：`PlayerViewController.swift`
- 位置：第 86 行附近
- 证据：`player = AVPlayer(url: videoURL!)`
- 问题说明：接口未返回播放地址时可能直接崩溃。
- 影响范围：播放器页面。
- 建议方案：使用 `guard let` 处理空值，并展示错误状态。
- 验收标准：
  - 不再使用该强制解包；
  - 空地址时展示可理解的错误状态；
  - 正常播放流程不受影响；
  - 相关构建或测试通过。
```

禁止仅输出以下空泛结论：

- 建议优化代码质量；
- 建议加强异常处理；
- 建议提升性能；
- 建议完善架构；
- 代码可读性有待提升。

每个问题必须包含可定位证据和可执行验收标准。

## 7. 待办事项管理

Review 发现的可执行事项统一同步到 `docs/todo.md`，建议使用以下格式：

```md
- [ ] R-002 修复播放器页面的强制解包
  - 优先级：高
  - 涉及文件：`PlayerViewController.swift`
  - 状态：待处理
  - 验收标准：
    - 不再使用相关强制解包；
    - 空值时展示错误状态；
    - 正常播放流程不受影响；
    - 相关构建或测试通过。
```

每个 Review 待办至少包含：

- 问题编号；
- 具体问题；
- 优先级；
- 涉及文件或模块；
- 当前状态；
- 验收标准。

不得只写“优化代码”“完善功能”等无法验收的事项。

## 8. 自动勾选规则

只有同时满足以下条件，才能把 `[ ]` 更新为 `[x]`：

1. 相关代码已经实际修改；
2. 对应验收标准已经满足；
3. 构建、测试、静态检查或必要人工验证已完成；
4. 原问题已无法复现，或对应风险已被消除；
5. 未引入明显的新问题。

状态规则：

- 仅修改代码但尚未验证：保持 `[ ]`，标记 `状态：已修改，等待验证`；
- 验证失败：保持 `[ ]`，记录失败原因；
- 当前环境无法验证：保持 `[ ]`，标记 `状态：等待人工验证`；
- 修改并验证通过：改为 `[x]`，记录完成依据。

不得为了提高完成率而删除、合并或隐藏未完成事项，也不得修改与当前 Review 无关的待办。

## 9. 修改边界

- Review 阶段优先分析，不直接进行大规模重构；
- 小型、明确、低风险的问题可以直接修复；
- 涉及架构、公共接口、数据结构或多个模块的修改，应先写入待办；
- 不得删除现有功能来规避问题；
- 不得因为测试困难而删除测试；
- 不得修改与当前问题无关的文件；
- 不得顺手格式化整个项目；
- 发现更优但影响较大的方案时，先说明收益、影响和风险，等待明确要求后再实施。

## 10. 验证要求

修复后根据项目实际情况执行：

1. 构建或编译；
2. 自动化测试；
3. 静态检查；
4. 功能验证；
5. 边界和异常状态验证；
6. `git diff` 检查。

验证规则：

- 实际执行并通过，才可以写“已通过”；
- 无法执行时，应写“未执行”并说明原因；
- 不得把“理论上可行”写成“验证通过”；
- 测试失败时不得删除测试或降低标准；
- 未通过验收标准的事项不得自动勾选。

## 11. Review 结束输出

```md
## 审查摘要

- 审查范围：
- 已检查文件：
- 已确认问题：
- 高概率问题：
- 待确认问题：
- 高优先级：
- 中优先级：
- 低优先级：
- 本次直接修复：
- 新增待办：
- 已完成待办：
- 等待验证：
- 构建状态：
- 测试状态：
- 静态检查状态：

## 问题详情

### R-001：问题标题

- 优先级：
- 可信度：
- 涉及文件：
- 证据：
- 影响：
- 建议：
- 验收标准：

## 待办变更

- 新增：
- 已完成：
- 等待验证：
- 未修改：

## 风险与限制

- 未验证内容：
- 缺失上下文：
- 潜在风险：
```

最后输出：

`REVIEW:v1`

## 12. 推荐工作流

```text
读取项目规则与上下文
    ↓
读取现有待办
    ↓
确定 Review 范围
    ↓
检查代码、配置、测试与差异
    ↓
输出有证据的问题
    ↓
标记优先级与可信度
    ↓
小问题直接修复
    ↓
复杂问题写入待办
    ↓
执行构建、测试和静态检查
    ↓
满足验收标准后自动勾选
    ↓
未满足则保留未完成状态并说明原因
    ↓
输出审查摘要
```
