## Context

`ReplyComposerSheet` 当前同时承担回复目标、富文本正文、附件、工具栏、快捷表情和表情选择入口。完整表情选择通过第二层 `NGAEmojiPickerSheet` 呈现，选择任意表情后立即关闭；分类使用只能容纳少量项目的 segmented picker。表情组、资源文件名推导和富文本恢复识别也位于同一个 Feature 文件中，目前只认识 `ng_`、`ac` 和 `a2_` 前缀。

ForumHub 已有 `PaperTheme` 和 `ForumGlass` 作为详情页视觉基础；NGA 回复提交已经稳定地把表情序列化为 `[img]URL[/img]`，图片附件则通过独立上传管线处理。新增目录必须保持这两个路径互不混淆，并遵守来源特有规则留在 NGA 适配边界的项目约束。

## Goals / Non-Goals

**Goals:**

- 在一个回复 Sheet 中形成稳定的标题、正文、附件、工具栏和可切换工具面板层级。
- 让用户连续浏览和插入六组 NGA 表情，不因单次选择丢失上下文。
- 以一个 NGA 来源专属目录定义分组、文件名、URL、显示信息和反向识别。
- 保证插入光标、富文本显示、文档恢复与最终提交序列化可逆。
- 复用项目现有主题和玻璃组件，并覆盖键盘、加载、提交和附件边界状态。

**Non-Goals:**

- 不改变 NGA 回复端点、认证、图片上传或回帖目标协议。
- 不把 NGA 表情暴露成 V2EX 或 LINUX DO 的共享能力。
- 不在本变更中切换到 `[s:group:name]` 提交格式。
- 不新增表情搜索、收藏、最近使用持久化或离线打包全部远程资源。
- 不重构详情页其他操作栏、分页或内容渲染架构。

## Decisions

### 1. 回复编辑器保持单 Sheet，并以显式工具模式驱动下方面板

回复 Sheet 保留一份正文和附件状态，增加仅属于展示层的输入模式，例如 `keyboard` 与 `emoji`。点击表情按钮时收起系统键盘并在工具栏下方展开内嵌目录；点击正文或再次点击表情按钮时恢复键盘模式。标题、回复目标、正文、附件、字数和发布按钮不随模式切换重建。

选择该方案是因为它消除嵌套 Sheet，并允许连续插入。保留独立表情 Sheet 的替代方案改动更小，但无法解决上下文割裂和单次选择即关闭的问题；直接把表情面板固定在键盘上方会在小屏幕上过度压缩正文，因此不采用。

### 2. 分类导航使用横向滚动标签，而不是 segmented picker

六个分类按 `NG娘、AC娘 v1、AC娘 v2、潘斯特、外域、企鹅` 排列。分类栏保持单行横向滚动，以主题强调色和下划线或等价的轻量选中态表达当前位置。表情网格使用稳定列数与点击尺寸，不为每个表情叠加高对比卡片。

系统 segmented picker 在六个中文分类下无法稳定容纳文字；多行按钮会增加垂直跳动，因此选择滚动标签。

### 3. 建立 NGA 来源专属的回复表情目录作为单一事实来源

目录负责提供组标识、标题、文件名序列、资源 URL 和从 URL 文件名反查条目的能力。Feature 只消费目录，不再自行以多个 `switch` 分散推导资源。

新增目录范围使用用户确认的数据：

| 分类 | 文件范围 | 数量 |
| --- | --- | ---: |
| 潘斯特 | `pt00.png...pt64.png` | 65 |
| 外域 | `dt01.png...dt33.png` | 33 |
| 企鹅 | `pg01.png...pg15.png` | 15 |

既有 NG娘、AC娘 v1、AC娘 v2 在本变更中保留当前可用集合，避免在缺少新的一手清单时顺手改变其边界。目录放在 NGA 数据/适配范围内，因为资源命名和 URL 是来源协议，不属于共享领域模型。

### 4. 保持 `[img]URL[/img]` 序列化，并扩展可逆识别

选中条目仍生成 `https://img4.nga.178.com/ngabbs/post/smile/<filename>`，提交文档仍输出 `[img]<url>[/img]`。`ReplyComposerEmoji` 的恢复不再只判断三种硬编码前缀，而是通过目录按文件名识别六组条目；无法识别的普通图片继续按普通正文或图片处理，不伪装成表情。

这一决定最大限度复用已验证的回复提交路径。改用 `[s:...]` 更紧凑，但需要补齐名称映射、服务端兼容和阅读解析证据，超出本变更范围。

### 5. 网络失败局部降级，不阻塞编辑和发布

表情缩略图加载失败时，条目显示稳定占位并允许后续重新加载；不得清空正文、关闭目录或阻止用户发布已有内容。目录边界由本地元数据确定，运行时不探测连续区间。

### 6. 快捷表情保持无持久化的确定性回退

键盘模式下可以保留现有快捷表情行，但必须从统一目录取得条目。此次不引入“最近使用”存储；如后续需要，可独立增加来源隔离的本地偏好而不改变目录协议。

### 7. 编辑器焦点使用一次性代次命令，Sheet 生命周期由固定大号面板承载

真机回归证明，持续焦点绑定会与 Sheet 关闭竞争：当 `updateUIView` 在 dismiss 过程中再次观察到“应该聚焦”，编辑器会重新调用 `becomeFirstResponder()`，使关闭动作长时间无法完成。焦点因此不再建模为持续布尔状态，而是建模为带递增 generation 的一次性命令；Coordinator 每代最多消费一次，并在文本视图挂入窗口后的下一主线程周期执行，避免在 SwiftUI 更新栈内同步改变 first responder。

切换到表情模式时，通过 responder chain 一次性请求当前 first responder 辞职，并在下一主线程周期切换面板；返回键盘时再签发新的焦点 generation。回复 Sheet 固定使用 `.large`，不再监听键盘高度动态计算 detent，避免键盘几何、编辑器焦点和 Sheet 高度形成相互反馈。关闭按钮只改变 Sheet 展示状态，回复目标统一在 `onDismiss` 清理，保证按钮关闭、下滑关闭和发布成功关闭遵循同一生命周期。

### 8. 发布任务可显式取消，回复网络步骤使用有限等待

真机人工验证发现，点击发布后真实 NGA 请求可能长时间等待。旧状态机会在 `isSubmittingReply` 为真时同时禁用发布、关闭按钮和交互式下滑，使用户无法退出。详情页现在持有发布 `Task`：重复发布、附件修改和交互式下滑仍在提交期间禁用，但关闭按钮成为明确的“取消发布并关闭”，先取消 Task 再退出 Sheet。主动取消静默结束，正文和附件保持原样，重新打开后可以继续编辑。

NGA 回复上下文 GET 与最终 POST 使用 30 秒请求超时，单个附件上传使用 60 秒超时。超时继续走现有 `ForumError.timeout` 用户反馈；取消通过 Swift Task 传播到 `URLSession.data(for:)`，不修改端点、认证字段、表单或上传协议。发布按钮同时显示“发布中”，避免只有小型转圈造成无反馈感。

## Risks / Trade-offs

- [内嵌表情面板与系统键盘切换可能造成焦点竞争或布局跳动] → 使用一次性焦点代次、异步 responder 切换、固定 `.large` detent 和稳定的编辑区/表情面板尺寸；小屏真机布局仍需独立验证。
- [113 个新增远程缩略图可能增加同时请求和滚动开销] → 继续使用懒加载网格和现有 `NGAImageLoader` 缓存，只加载可见条目，不预取整个目录。
- [远程资源未来移除或改名] → 本地目录保持可测试，单项失败局部降级；不通过运行时网络扫描改变目录。
- [`[img]` 表情在阅读端仍表现为图片语义而非命名表情] → 接受现有兼容行为；切换 `[s:...]` 另立有协议证据的变更。
- [把目录移出 Feature 可能扩大文件变动] → 只提取来源专属数据与识别，不顺手重构富文本编辑器或通用内容模型。
- [真实回复请求停滞会把用户锁在提交态] → 显式保留可取消关闭路径，并为回复上下文、附件上传和最终发布设置有限等待；取消保留草稿且不显示失败。

## Migration Plan

1. 先增加统一目录和边界/反向识别测试，保持现有三组 UI 行为不变。
2. 让 `ReplyComposerEmoji` 与快捷栏改用统一目录，并验证旧文档序列化往返不回归。
3. 将嵌套表情 Sheet 替换为单 Sheet 内嵌面板，增加六组分类和连续选择。
4. 完成真机构建、聚焦测试及键盘/表情/附件/提交人工验证后更新相关中文文档。
5. 若出现严重回归，可先恢复旧表情展示入口；目录与兼容识别可独立保留，不影响提交协议。
